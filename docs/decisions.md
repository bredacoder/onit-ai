# Technical decisions — Onit M0 (ADRs)

> Stack decisions locked before the foundation. Each one: **Decision · Why ·
> Rejected alternative.** They complement what the PRD already fixed (Go,
> `anthropic-sdk-go` behind `PortLLM`, Postgres+pgvector, hexagonal, git-style
> CLI, single binary, `Negotiation` FSM). Date: 2026-05-29.

## ADR-001 — CLI library: `cobra`
- **Decision:** use `spf13/cobra` for the git-style subcommands (`onit new`, `onit tasks`, `onit task <id>`, `onit reply <id>`, `onit feedback`).
- **Why:** subcommands, help, and flags out of the box; de facto standard (`gh`, `kubectl`); a single well-supported dependency — aligned with "idiomatic, few deps" (PRD 9.4).
- **Rejected alternative:** stdlib `flag` (rewriting routing by hand); `urfave/cli` (no clear advantage).

## ADR-002 — Database access: `pgx` + `sqlc`
- **Decision:** `pgx` driver; queries in plain SQL with type-safe Go code generated by `sqlc`.
- **Why:** readable SQL, type-safe Go, no ORM magic nor runtime reflection; every query gets reviewed. This is "more own code over a Postgres client, no framework" (PRD 9.4). `pgx` supports pgvector.
- **Rejected alternative:** raw `pgx` (repetitive boilerplate); `sqlx` (fewer compile-time guarantees); `gorm`/`ent` ORM (against the spirit of the PRD).

## ADR-003 — Agent loop: thin loop in the core, SDK only for transport
- **Decision:** `PortLLM` is a **single-turn** primitive (`Complete(req) -> (text | tool_calls)`). The agent loop (`while there is tool_use { run tools; resend }`) is written **once in the core**, provider-agnostic. The `anthropic-sdk-go` is used **only for transport** (the `Messages.New` call, auth, retries) inside the adapter — the toolrunner is **not** used.
- **Why:** keeps orchestration in the core (principle of section 9.1), does not reimplement transport (PRD 9.4), and makes multi-provider trivial — adding OpenAI/Gemini is a thin adapter translating `Complete`, with the loop untouched. Zero loop reimplementation in the future. Fits the learning goal (PRD 9.4) and reviewing/understanding every line.
- **Rejected alternative:** the SDK toolrunner — faster in M0, but it is a loop coupled to Anthropic *below* the port, which would force duplicating the loop logic per provider or a refactor later. The cost of our own loop (~40 lines you understand) does not justify deferring.

## ADR-004 — Credentials at-rest: app-level AES-GCM, key via env
- **Decision:** tokens (e.g., Google OAuth) stored in a per-`User` table, in a column encrypted with AES-GCM (stdlib `crypto`); key kept outside the database, injected via env. System secrets (API keys, DB URL) in env/`.env`, not committed.
- **Why:** satisfies "multi-tenant-ready always" (PRD 8) without infra too early; the M2 KMS is just swapping the key provider.
- **Rejected alternative:** plaintext (creates debt, contradicts PRD 8); external KMS now (infra/cost too early for M0 single-user).

## ADR-005 — Migrations: `goose`
- **Decision:** `goose` with up/down SQL files.
- **Why:** simple, plain SQL, a single binary, no DSL — aligned with "simple".
- **Rejected alternative:** `golang-migrate` (equivalent); `atlas` (HCL/extra concept).

## ADR-006 — Layout: interfaces (ports) in the core package
- **Decision:** `/cmd/onit` (CLI entry), `/internal/core` (pure core + ports declared here), `/internal/adapters/{postgres,anthropic,places,gcal,memory}`. Adapters import the core; the core imports nobody.
- **Why:** enforces the dependency rule "everything points inward" (PRD 9.1); Go idiom "accept interfaces, return structs".
- **Rejected alternative:** a separate `ports` package (interfaces far from their consumers, less idiomatic).

## ADR-007 — Local dev: `docker-compose` + `Makefile`
- **Decision:** `docker-compose` with the `pgvector/pgvector` image; `Makefile` with `db-up`, `migrate`, `test`, etc.
- **Why:** reproducible Postgres+pgvector in one command; Makefile standardizes the workflow.
- **Rejected alternative:** Postgres installed on the machine (not reproducible); Taskfile (preference, no real gain).

## ADR-008 — Port location: cross-cutting ports in a thin root `core` package
- **Decision:** context-specific ports stay in their bounded-context package (`discovery.Source`/`discovery.Port`, `memory.Port`, `scheduling.CalendarPort`); the cross-cutting ports used by several core packages (`PortLLM`, `PortPersistence`, `PortClock`) are declared in a thin root `core` package. This complements ADR-006 (layout) and the domain map in `docs/bounded-contexts.md`.
- **Why:** keeps the ADR-006 rule "interface near its consumer" for single-consumer ports, but a port consumed identically by `understanding`, `negotiation`, and `agent` has no single natural home — declaring it in a thin root package avoids import cycles between contexts and a duplicated interface per consumer.
- **Rejected alternative:** (a) declare the cross-cutting ports in the `agent` package (primary loop consumer) — risks `understanding`/`negotiation` importing `agent` just for the port, inverting the dependency; a separate `ports/` package — already rejected by ADR-006 (interfaces far from consumers).
